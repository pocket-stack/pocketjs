Pocket3D The scriptable 3D runtime

Native 3Dscripted in TypeScript

Pocket3D is a small, fast Rust renderer for 3D worlds — and the same PocketJS stack that runs on a PSP renders your game's UI on top of it. Cores are native; gameplay and overlays are JavaScript. OpenStrike, a CS-like FPS on classic BSP maps, is the proof.

Explore OpenStrike How it works
The overlay is a PocketJS app

Your HUD is JSX, not engine code.

Every heads-up element in OpenStrike — crosshair, health, ammo, kill feed, round banners, the damage flash — is a Solid component in the PocketJS framework, the very same stack that renders UI on a Sony PSP. Pocket3D composites it over the 3D frame in one pass.

So a 3D app gets a real, reactive UI layer for free: you author overlays as components with Tailwind classes and signals, and the native side handles the pixels. No immediate-mode bitmap fiddling, no bespoke text renderer.

  • Solid / Vue Vapor components, Tailwind styling, native animation.
  • Composited over the scene — the same UI runtime, from 3D games to PSP handhelds.

A tiny renderer with a scripting seam

Low-cost 3D, freely modded.

Pocket3D is a lean wgpu forward renderer: GoldSrc BSP worlds are a first-class format — lightmaps, hull-trace collision, skeletal animation — loaded and drawn without a heavyweight engine. The whole runtime is small enough to read in an afternoon.

On top of it sits a QuickJS mod runtime. A Rust core owns simulation and rendering for speed; a JavaScript guest owns the product for freedom. In OpenStrike the round flow, scoring and weapon tables are all TypeScript — change a number, rebuild the bundle, and you've made a mod. This scripting seam is meant to generalize to any kind of 3D app.

  • BSP maps, lightmaps and clipnode collision as native data — no physics engine.
  • Gameplay as TypeScript mods — apps, games and mods are the same kind of artifact.
